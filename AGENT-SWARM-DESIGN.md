## 🔄 8 步工作流程（实战版）

### Step 1: 需求 → 麦克虾范围界定
```
场景：老板在 Telegram 说"加个功能"

麦克虾行动：
1. 从 MEMORY.md 读取相关上下文
2. 理解需求的业务背景
3. 拆解任务
4. 选择合适的代理
```

### Step 2: 生成代理
```bash
# 每个代理独立工作区
git worktree add ../feature-branch-1 origin/main
tmux new-session -d -s agent-1 "node agent-runner.js --task='...' --model=codex"
```

### Step 3: 监控循环
```bash
# cron 每 10 分钟检查
*/10 * * * * cd ~/.openclaw && node monitor-agents.js
```

检查内容：
- ✅ tmux 会话是否存活
- ✅ 是否有新 PR
- ✅ CI 状态
- ✅ 失败自动重启（最多 3 次）
- ⚠️ 需要人工介入 → Telegram 通知老板

### Step 4: 代理创建 PR
```bash
# 代理完成工作后
git commit -m "feat: ..."
git push origin feature-branch-1
gh pr create --fill
```

### Step 5: 自动化代码审查
```
三个 AI 模型审查：

1. Codex Reviewer
   - 擅长：边界情况、逻辑错误、竞态条件
   - 准确率：高，误报率低

2. Gemini Code Assist
   - 擅长：安全问题、可扩展性
   - 优势：免费，建议具体修复方案

3. Claude Code Reviewer
   - 擅长：过度工程警告
   - 用法：只看 critical 级别
```

### Step 6: 自动化测试
```yaml
CI Pipeline:
  - Lint + TypeScript 检查
  - 单元测试
  - E2E 测试
  - Playwright 测试（预览环境）
  - UI 变更必须有截图（否则 CI 失败）
```

### Step 7: 人工审查（老板）
```
Telegram 通知：
"🦐 PR #341 准备审查"

此时状态：
✅ CI 通过
✅ 3 个 AI 审查通过
✅ 截图已附
✅ 边界情况已记录

老板审查时间：5-10 分钟
很多 PR 看截图就能合并，不用读代码
```

### Step 8: 合并
```bash
gh pr merge --merge --delete-branch
# 每日 cron 清理 orphaned worktrees
```

## 🧠 Ralph Loop V2 - 智能重试机制

**传统 Ralph Loop:** 同样的 prompt 循环运行

**麦克虾的 V2:** 失败时智能调整

```
代理失败 → 麦克虾分析失败原因 → 改进 prompt → 重启代理

例子：
❌ 代理：上下文不足
✅ 麦克虾："只关注这 3 个文件，这是客户的具体需求..."

❌ 代理：方向错误
✅ 麦克虾："停，客户要的是 X 不是 Y，这是会议记录..."

❌ 代理：需要澄清
✅ 麦克虾："这是客户的邮件和他们公司的业务说明..."
```

### 主动寻找工作
```
早晨:
  扫描错误日志 → 发现 4 个新错误 → 生成 4 个代理修复

会议后:
  扫描会议笔记 → 标记 3 个功能需求 → 生成 3 个代理实现

晚上:
  扫描 git log → 生成 Claude Code 更新 changelog 和文档
```

## 💡 关键洞察

### 为什么需要编排层？
```
❌ Codex/Claude Code 只有代码上下文
✅ OpenClaw 有完整业务上下文

- 客户数据
- 会议笔记
- 历史决策
- 什么成功过/失败过

编排层翻译业务上下文 → 精准 agent prompts
```

### 上下文窗口是零和游戏
```
填满代码 → 没空间放业务上下文
填满客户历史 → 没空间放代码库

两层系统解决：
- 编码代理：只加载代码
- 编排器：只加载业务上下文
```
